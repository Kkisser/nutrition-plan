# Схема базы данных (Задача Б3)

**Дата:** 18 мая 2026
**Назначение:** логическая схема БД для разработки ядра. Опирается на матмодель v2, контракт API Б4, функционал v4, обновлённые DFD-хранилища пользователя (D1–D6). Заменяет устаревшую структуру раздела 6 диплома (которая была под бюджет/магазины/5 уровней активности/пользовательские рецепты).

**СУБД:** PostgreSQL (ядро). price-service отдельный stateless-сервис, БД не использует, цены НЕ хранятся в этой базе — они берутся по запросу из Edadil в момент расчёта вилки и нигде в модуле не сохраняются.

**Обозначения:** PK — первичный ключ, FK — внешний ключ, UQ — уникальное ограничение. Связи — нотация «вороньей лапки» (crow's foot). Имена на английском, русский перевод в скобках.

---

## 1. Что НЕ входит в схему (осознанно, по финальным решениям)

- Таблицы магазинов, цен, выбранных магазинов (`SHOPS`, `PRODUCT_PRICES`, `USER_SHOPS`) — цены не хранятся, price-service stateless (Решение №6).
- Поле/таблица бюджета — бюджет убран (замечания научрука).
- Признак пропуска приёма пищи — пропуск убран (Решение функц.).
- Поле «уровень активности из 5 категорий» — заменено на группу КФА I–IV (Решение №14).
- `source='user'` у рецептов / пользовательские рецепты — каталог только курируемый (Решение №2).
- Лайки/дизлайки блюд и продуктов — оставлено только исключение блюд (Решение №11).
- Отчёт по микронутриентам для пользователя — недобор только во внутренней таблице (Решение №13).

---

## 2. Кластеры

Схема логически делится на три кластера: пользовательский, рецептурно-продуктовый, плановый.

## 4. Рецептурно-продуктовый кластер

### PRODUCTS (продукты)
| Поле | Тип | Ключ | Комментарий |
|---|---|---|---|
| product_id | uuid | PK | |
| name | varchar | UQ | наименование |
| category | varchar | | категория |
| kcal_100 | numeric | | энергетическая ценность на 100 г |
| protein_100 | numeric | | белки на 100 г |
| fat_100 | numeric | | жиры на 100 г |
| carb_100 | numeric | | углеводы на 100 г |
| default_unit | enum(g,ml,pcs) | | единица по умолчанию (совместима с price-service) |

Источник наполнения — справочник Скурихина-Тутельяна (Решение №2 журнала).

### PRODUCT_MICRONUTRIENTS (микронутриенты продуктов)
| Поле | Тип | Ключ | Комментарий |
|---|---|---|---|
| product_id | uuid | PK, FK→PRODUCTS | |
| nutrient_id | varchar | PK, FK→MICRONUTRIENTS | код микронутриента |
| amount_100 | numeric | | содержание на 100 г |

Вынесено в отдельную таблицу: ~29 микронутриентов на продукт (Решение №1).

### MICRONUTRIENTS (справочник микронутриентов)
| Поле | Тип | Ключ | Комментарий |
|---|---|---|---|
| nutrient_id | varchar | PK | код (напр. vit_d, ca, fe) |
| name | varchar | | наименование |
| norm_unit | varchar | | единица нормы (мг, мкг, МЕ) |
| ul_value | numeric | | верхний допустимый уровень (МР 2.3.1.1915-04; для непокрытых — EFSA), nullable |

Нормы по полу/возрасту — см. MICRONUTRIENT_NORMS.

### MICRONUTRIENT_NORMS (нормы микронутриентов по группам)
| Поле | Тип | Ключ | Комментарий |
|---|---|---|---|

| Поле | Тип | Ключ | Комментарий |
|---|---|---|---|

| age_group | enum | PK | возрастные группы МР |
| kfa_group | enum(I,II,III,IV) | PK | группа КФА |
| kcal_norm | numeric | | норма энергии из таблиц МР 2.3.1.0253-21 |

| fat_g_norm | numeric | | |

Ключевая таблица Решения №14: энергонорма берётся отсюда по (пол, возраст, группа КФА), без формулы Миффлина.

### RECIPES (рецепты / блюда)
| Поле | Тип | Ключ | Комментарий |
|---|---|---|---|
| recipe_id | uuid | PK | |
| name | varchar | | наименование |
| instruction | text | | пошаговая инструкция |
| cook_time_min | int | | время приготовления |
| base_portions | int | | базовое число порций |

Поле `source='user'` убрано — каталог только курируемый администратором (Решение №2). UQ по external_id (без пары с source).

### RECIPE_INGREDIENTS (состав блюд)
| Поле | Тип | Ключ | Комментарий |
|---|---|---|---|
| recipe_id | uuid | PK, FK→RECIPES | |
| product_id | uuid | PK, FK→PRODUCTS | |
| amount | numeric | | количество продукта |
| unit | enum(g,ml,pcs) | | единица измерения |

Связь RECIPES ∞—∞ PRODUCTS. КБЖУ и микронутриенты блюда вычисляются как взвешенная сумма по продуктам (матмодель v2, раздел 1.3).

### RECIPE_DIET_COMPAT (совместимость блюд с диетами)

|---|---|---|---|
| recipe_id | uuid | PK, FK→RECIPES | |
| diet_id | enum | PK, FK→DIETS | допустимые типы питания для блюда |

### RECIPE_ALLERGENS (аллергены блюда)
| Поле | Тип | Ключ |
|---|---|---|
| recipe_id | uuid | PK, FK→RECIPES |
| allergen | enum(...) | PK |

Для фильтрации блюд по аллергиям пользователя (блок 4 алгоритма).

## 5. Плановый кластер

|---|---|---|---|
| plan_id | uuid | PK | |
| user_id | uuid | FK→USERS | |
| week_ref | varchar | | ISO-неделя (напр. 2026-W20) |
| date_start | date | | |
| date_end | date | | |
| created_at | timestamp | | |

Поле бюджета убрано (замечания научрука).

### MEAL_PLAN_SLOTS (позиции плана)
| Поле | Тип | Ключ | Комментарий |
|---|---|---|---|
| slot_id | uuid | PK | |
| plan_id | uuid | FK→MEAL_PLANS | |
| day_no | int | | день 1..7 |
| meal_type | enum(breakfast,lunch,dinner,snack) | | приём пищи |

| is_pinned | boolean | | закреплено пользователем (Решение №4) |

Признак пропуска убран (пропуск приёма пищи отменён). «Слот» — внутреннее имя таблицы; в пользовательском тексте/UI — «приём пищи».

### MICRONUTRIENT_CARRYOVER (перенос недобора микронутриентов)
| Поле | Тип | Ключ | Комментарий |
|---|---|---|---|
| user_id | uuid | PK, FK→USERS | |
| week_ref | varchar | PK | неделя, по итогам которой посчитан недобор |
| nutrient_id | varchar | PK, FK→MICRONUTRIENTS | |
| deficit_per_day | numeric | | недобор в пересчёте на сутки (Δ_i) |

Реализация Решения №1: по итогам недели считается недобор, на следующей неделе целевое значение повышается на deficit/7, но не выше UL. Не накапливается между итерациями. Пользователю не показывается (Решение №13).

### SHOPPING_LISTS (списки покупок)
| Поле | Тип | Ключ | Комментарий |
|---|---|---|---|
| list_id | uuid | PK | |
| plan_id | uuid | FK→MEAL_PLANS, UQ | один план — не более одного списка (1—0..1) |
| user_id | uuid | FK→USERS | владелец |

Итоговой стоимости в заголовке нет — цена не хранится (считается по запросу через price-service).

### SHOPPING_LIST_ITEMS (позиции списка покупок)
| Поле | Тип | Ключ | Комментарий |
|---|---|---|---|
| list_id | uuid | PK, FK→SHOPPING_LISTS | |
| product_name | varchar | PK | агрегированное наименование |
| amount | numeric | | количество (уже × persons) |
| unit | enum(g,ml,pcs) | | единица (совместима с price-service) |
| is_purchased | boolean | | отметка приобретения |
| is_manual | boolean | | добавлено пользователем вручную |

Поля цены и магазина убраны. Формат (product_name, amount, unit) совпадает с входом price-service `items[]`.

### PLAN_MEMBERS (участники объединения меню)
| Поле | Тип | Ключ | Комментарий |
|---|---|---|---|
| list_id | uuid | PK, FK→SHOPPING_LISTS | |
| member_user_id | uuid | PK, FK→USERS | |
| joined_at | timestamp | | |

Реализация кнопки «объединить меню» (Решение функц. №1): одноимённые продукты двух пользователей суммируются в сводный список. Объём взаимодействия ограничен объединением списков (раздел 5 ВКР).

## 7. Как схема обслуживает алгоритм (связь с блок-схемой)

| Блок алгоритма | Таблицы |
|---|---|
| Б1 приём входных данных | USERS, USER_ALLERGIES, USER_EXCLUDED_PRODUCTS, USER_DISH_EXCLUSIONS, ACTIVITY_SURVEY |
| Б2 целевые показатели | ENERGY_NORMS → USER_GOALS |
| Б3 распределение, закрепление | MEAL_PLAN_SLOTS (is_pinned) |
| Б4 фильтрация каталога | RECIPES, RECIPE_DIET_COMPAT, RECIPE_ALLERGENS, RECIPE_INGREDIENTS, USER_DISH_EXCLUSIONS |
| Б5–Б6 подбор и балансировка | RECIPES, RECIPE_INGREDIENTS, PRODUCTS |
| Б7 недобор микронутриентов | PRODUCT_MICRONUTRIENTS, MICRONUTRIENT_NORMS, MICRONUTRIENT_CARRYOVER |
| Б10 список покупок | SHOPPING_LISTS, SHOPPING_LIST_ITEMS, PLAN_MEMBERS |
| Б11–Б13 цена | (БД не участвует — price-service по API Edadil) |

## 8. Открытые технические мелочи (не блокеры)

- Возрастные группы (`age_group`) — перечислить точные границы по МР 2.3.1.0253-21 при наполнении ENERGY_NORMS.
- Коды `nutrient_id` — завести справочник MICRONUTRIENTS при загрузке данных Скурихина-Тутельяна (задача П1/П4).
- Индексы и ограничения целостности — на этапе физической модели (DDL), здесь логический уровень.
