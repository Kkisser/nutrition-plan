# Контракт API — граница «вход → ядро → выход» (Задача Б4)

**Дата:** 18 мая 2026
**Назначение:** зафиксировать границы между подсистемами, чтобы Филин и Аловердян могли вести разработку параллельно. Опирается на матмодель v2, блок-схему алгоритма и функционал v4.
**Статус:** актуальный контракт. price-service доработан (вилка цены + выбор магазина реализованы). Контракт ядра — проектный, реализуется при разработке.

---

## 1. Общая картина потоков

```
Пользователь → [Опросник/профиль] → Профиль (БД)
                                         │
                                         ▼
                          ┌──────────────────────────────┐
                          │  ЯДРО (сервер приложений, Go) │
                          │  формирование плана питания   │
                          └──────────────────────────────┘
                                         │
                          план + агрегированный список покупок
                                         │
                          ┌──────────────┴───────────────┐
                          ▼                               ▼
                  [Возврат пользователю]      [price-service, опционально]
                   план + список покупок       вилка [мин; макс] по магазину
```

Ядро и price-service — **разные подсистемы**. Ядро формирует план и список покупок. price-service только оценивает стоимость уже готового списка и только если пользователь выбрал магазин. price-service stateless, PostgreSQL не использует, цены не хранит.

---

## 2. Контракт ЯДРА: вход → формирование плана → выход

### 2.1. Вход ядра (что подаётся на формирование плана)

Логическая структура запроса на формирование недельного плана:

```json
{
  "user_id": "uuid",
  "profile": {
    "sex": "male | female",
    "age": 30,
    "height_cm": 180,
    "weight_kg": 80,
    "kfa_group": "I | II | III | IV",
    "goal": "deficit | maintain | surplus",
    "diet_type": "classic | keto | vegetarian | vegan | paleo | fasting",
    "allergens": ["milk", "eggs", "..."],
    "excluded_products": ["лук", "..."],
    "meals": ["breakfast", "lunch", "dinner", "snack"],
    "persons": 1
  },
  "manual_targets_override": {
    "enabled": false,
    "kcal": null, "protein_g": null, "fat_g": null, "carb_g": null
  },
  "pinned_dishes": [
    { "day": 1, "meal": "breakfast", "dish_id": "uuid" }
  ],
  "excluded_dishes": ["uuid", "..."],
  "micronutrient_carryover": {
    "week_ref": "2026-W19",
    "deficits": [ { "nutrient_id": "vit_d", "deficit_per_day": 4.1 } ]
  }
}
```

Пояснения по полям:
- `kfa_group` — результат мини-анкеты активности (4 группы КФА по МР 2.3.1.0253-21), не сырые ответы анкеты. Маппинг ответов анкеты в группу выполняется до вызова ядра.
- `manual_targets_override` — ручная корректировка целевых КБЖУ; значения должны быть в пределах ±15% от нормы МР, проверка границ — на стороне ядра (значение зажимается на границе коридора).
- `pinned_dishes` — закреплённые блюда (Решение №4); ядро их не подбирает, вычитает из дневного целевого числа.
- `excluded_dishes` — список исключённых пользователем блюд (Решение №11), таблица `user_dish_exclusions`.
- `micronutrient_carryover` — недобор микронутриентов за предыдущую неделю (Решение №1); при первой генерации пустой.

### 2.2. Выход ядра (что возвращается)

```json
{
  "user_id": "uuid",
  "week_ref": "2026-W20",
  "plan": [
    {
      "day": 1,
      "meals": [
        {
          "meal": "breakfast",
          "dish_id": "uuid",
          "dish_title": "string",
          "portions": 1.0,
          "pinned": false,
          "kcal": 525, "protein_g": 30, "fat_g": 18, "carb_g": 60
        }
      ],
      "day_totals": { "kcal": 2100, "protein_g": 90, "fat_g": 70, "carb_g": 260 }
    }
  ],
  "shopping_list": [
    { "ingredient_name": "молоко", "amount": 2000, "unit": "ml" },
    { "ingredient_name": "куриное филе", "amount": 1400, "unit": "g" }
  ],
  "compliance": {
    "in_corridor": true,
    "violations": [],
    "message": null
  },
  "micronutrient_carryover_next": {
    "week_ref": "2026-W20",
    "deficits": [ { "nutrient_id": "vit_d", "deficit_per_day": 3.2 } ]
  }
}
```

Пояснения:
- `shopping_list` — агрегированный список: одноимённые продукты суммированы, количество уже умножено на `persons`. Это ровно тот формат, который принимает price-service (`items[]`: `ingredient_name`, `amount`, `unit`).
- `compliance` — результат проверки коридора (блок 8 алгоритма). Если `in_corridor=false`, `violations` перечисляет показатели нарушения, `message` — текст для пользователя; фронт показывает предупреждение и варианты действий (снять закрепления / принять с отклонением / переформировать) — Решение №4.
- `micronutrient_carryover_next` — рассчитанный недобор для переноса на следующую неделю; сохраняется в БД, в UI пользователю не показывается (Решение №13).

### 2.3. Граница ответственности

- Маппинг ответов мини-анкеты → `kfa_group`: **до ядра** (подсистема профиля).
- Расчёт целевых КБЖУ из нормы МР + поправка цели: **внутри ядра**.
- Проверка коридора и расчёт недобора: **внутри ядра** (блоки 7–8).
- Стоимостная оценка: **вне ядра** (price-service), вызывается отдельно и опционально.

## 3. Контракт price-service (стоимостная оценка)

Источник: документация price-service `backend-integration-guide.md` и `backend-response-format.md` (актуальные, price-service доработан). Здесь — то, что важно для контракта с ядром.

### 3.1. Актуальное состояние (price-service ДОРАБОТАН — задача П2 закрыта)

price-service уже реализует вилку цены и режим выбранного магазина. Доработка под Решение №6 выполнена; это больше не «дельта», а факт.

- `POST http://localhost:8085/estimate`, вход — агрегированный `items[]` (формат совпадает с `shopping_list` ядра).
- Возвращает **вилку**: `min_total_price`, `max_total_price`, объект `price_range {min_price, max_price}`. Поле `total_price` оставлено для обратной совместимости и равно `min_total_price`.
- Stateless, PostgreSQL/Docker не использует, бизнес-данные не хранит.
- `X-Request-ID` — бэкенд передаёт свой correlation id, возвращается в `request_id`.

### 3.2. Запрос к price-service

```http
POST /estimate
Content-Type: application/json
X-Request-ID: backend-correlation-id
```
```json
{
  "selected_shop_uuid": "052761c1-6775-4ac4-8d9d-2f03c974932b",
  "items": [
    { "ingredient_name": "молоко", "amount": 2000, "unit": "ml" }
  ]
}
```

Ключевое уточнение по выбору магазина (ИСПРАВЛЕНИЕ относительно ранней версии контракта, где предполагался `retailer_slug`):
- Магазин задаётся **не `retailer_slug`**, а **`selected_shop_uuid`** — Edadil `shopUuid` конкретной торговой точки сети Пятёрочка (из `partner.nearest[]`). Alias `shop_uuid`; query-параметры `?selected_shop_uuid=...` / `?shop_uuid=...` тоже поддерживаются.
- `selected_shop_uuid` **необязателен**:
  - не передан → `pricing_scope = "nearest_shops_range"`, вилка по ближайшим магазинам Edadil (min из нижних цен, max из верхних);
  - передан → `pricing_scope = "selected_shop"`, цены только по этому магазину, `min_total_price` ≈ `max_total_price`.
- Пользователь по сценарию приложения пропустил оценку (блок 11→12 алгоритма) → бэкенд **не вызывает** price-service, вилка не формируется.
- `retailer_slug` всегда `5ka` (Пятёрочка), задаётся конфигурацией сервиса, а не запросом.

### 3.3a. Ответ price-service (фактический)

```json
{
  "request_id": "backend-correlation-id",
  "status": "ok | partial | failed",
  "calculated_at": "2026-05-10T18:01:29Z",
  "is_fully_priced": true,
  "price_type": "estimated_reference_price",
  "pricing_scope": "nearest_shops_range | selected_shop",
  "selected_shop_uuid": "... если передан",
  "retailer": "Пятёрочка",
  "retailer_slug": "5ka",
  "currency": "RUB",
  "total_price": 273.98,
  "min_total_price": 273.98,
  "max_total_price": 309.98,
  "price_range": { "min_price": 273.98, "max_price": 309.98 },
  "priced_items_count": 3,
  "unpriced_items_count": 0,
  "items": [ "...selected_option.packages[] с unit_price_min/max, subtotal_min/max..." ],
  "unpriced_items": [ ]
}
```

Как бэкенд читает результат:
- Вилка корзины — `min_total_price` / `max_total_price` (или `price_range`). `total_price` — только совместимое поле нижней цены.
- `status = "partial"` — НЕ транспортная ошибка, а нормальный бизнес-результат (часть позиций не нашлась/не подошла по фасовке). Показать `unpriced_items[]` как предупреждение.
- Выбранные товары — `items[].selected_option.packages[]` (только целые упаковки; дробную цену не считать).
- HTTP 200 даже при ошибке отдельных позиций (причина в `items[].status`, `error_message`, `unpriced_items[]`). HTTP 400 — структурно битый запрос; 503 — Edadil недоступен полностью.
- Новый статус позиции: `selected_shop_price_not_found` (товар есть, цены в выбранном магазине нет).

Откуда брать `selected_shop_uuid`: Edadil `shopUuid` из `partner.nearest[]`. Отдельного endpoint выбора магазина в MVP нет; для разработки виден через `POST /estimate?debug=true` в `items[].debug.nearest_shops`. В полноценном приложении бэкенд передаёт уже выбранный пользователем `shopUuid`.

Что бэкенду сохранять в своей БД (price-service не хранит): `shopping_list_id`, `request_id`, `calculated_at`, `pricing_scope`, `selected_shop_uuid`, `min_total_price`, `max_total_price`, `priced/unpriced_items_count`, `is_fully_priced`, сырой JSON, `items[].selected_option.packages[]`, `unpriced_items[]`. Цены как справочник в БД модуля не хранятся (Решение №6) — сохраняется только результат конкретного расчёта при необходимости истории.

### 3.3. Граница ответственности price-service

- Принимает **уже агрегированный** список (агрегацию делает ядро, не price-service).
- Не формирует план, не работает с КБЖУ, не проверяет диеты/аллергии.
- Не хранит данные (stateless), не использует PostgreSQL.
- Возвращает только стоимостную вилку и детализацию по позициям.

## 4. Сводка точек интеграции

| Граница | Кто вызывает | Что передаёт | Что получает |
|---|---|---|---|
| Профиль → Ядро | бэкенд | профиль + закрепления + исключения + недобор | план + список покупок + compliance |
| Ядро → price-service | бэкенд (опционально) | агрегированный список + `selected_shop_uuid` (опц.) | `min_total_price` + `max_total_price` + `price_range` |
| Ядро → БД | ядро | план, недобор для переноса | — |
| price-service → Edadil | price-service | поисковые запросы по товарам | цены, фасовки |

## 5. Что НЕ входит в контракт (осознанно)

- Бюджет и стоимостная оптимизация — отсутствуют (Решение №6, замечания научрука).
- Несколько магазинов / распределение покупок — отсутствует.
- Проверка фактического наличия товара, геопривязка — отсутствует.
- Отчёт по микронутриентам в выходе для UI — отсутствует (Решение №13); недобор только во внутренней структуре.
- Отметки пользователя о реально съеденном — отсутствуют (Решение №1, фактическое = запланированное).

## 6. Открытые технические вопросы (не блокеры контракта)

- Конкретные коды единиц измерения в `shopping_list` ограничены допустимыми price-service: `g`, `ml`, `pcs`. Ядро при агрегации должно приводить количества к этим трём единицам — правило приведения зафиксировать при реализации (задача уровня кода, не контракта).
- Формат `week_ref` (`2026-W20`, ISO-неделя) — принять как соглашение; при реализации сверить с тем, как недели хранятся в БД (Задача Б3).
- Сопоставление `nutrient_id` (строковые коды микронутриентов) с перечнем МР 2.3.1.0253-21 — справочник кодов завести при проектировании БД (Задача Б3).
