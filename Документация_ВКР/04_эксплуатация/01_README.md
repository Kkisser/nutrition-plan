# Проект: программный модуль формирования персонализированного плана питания

Документы для разработки ядра. Все решения финализированы, 4 блокера закрыты.
Назначение этой папки — спецификации, по которым пишется код. Это не текст диплома.

## Порядок чтения (для нового человека или для Claude Code)

1. **ФУНКЦИОНАЛ.md** — что приложение должно делать (функциональное описание v4).
2. **МАТМОДЕЛЬ.txt** — как считается план: штрафная функция, нормы, алгоритм. Первоисточник.
3. **АЛГОРИТМ_блок-схема.svg** — пошаговая логика формирования плана (открыть в браузере).
4. **СХЕМА_БД.md** — таблицы, поля, ключи, связи. По ней создаётся база.
5. **КОНТРАКТ_API.md** — что подаётся в ядро, что оно возвращает, стык с price-service.
6. **ТРЕКЕР_задач.md** — что сделано, что осталось (параллельные задачи и текст ВКР).

## Технологический стек (зафиксирован)

- Ядро формирования плана — Go
- Загрузка справочных данных (Скурихин-Тутельян) — Python
- Хранение — PostgreSQL
- Кэш — Redis
- Фронтенд — React / TypeScript
- price-service — отдельный Go-сервис (уже работает, см. ниже)

## Что уже готово вне этой папки

- **price-service** — отдельный Go-сервис, считает цены через Edadil API по сети Пятёрочка.
  Уже работает. Требует доработки: вернуть `min_total_price` + `max_total_price` вместо
  одного `total_price` и принимать поле `retailer_slug` (см. КОНТРАКТ_API.md, раздел 3.2).
  Документация price-service — у Филина/Аловердяна (README price-service,
  «формат ответа сервиса для бека»).

## Ключевые принципы (чтобы не отклониться)

- Энергонорма берётся из таблиц МР 2.3.1.0253-21 по (пол, возраст, группа КФА).
  Формула Миффлина — Сан Жеора НЕ используется.
- Бюджет, выбор магазинов как параметр, проверка наличия товаров — НЕ реализуются.
- Цена — опциональная вилка [мин; макс] по одному магазину, постфактум, через price-service.
  Цены в БД НЕ хранятся.
- Микронутриенты НЕ входят в формулу F — отдельный механизм переноса недобора на
  следующую неделю (см. МАТМОДЕЛЬ.txt раздел 7).
- Пользовательские рецепты, лайки/дизлайки, пропуск приёма пищи — НЕ реализуются.
- Каталог блюд — только курируемый администратором.

## Очерёдность разработки (рекомендация)

1. Создать схему БД (PostgreSQL DDL по СХЕМА_БД.md).
2. Загрузчик справочника продуктов Скурихина-Тутельяна (Python) → таблицы PRODUCTS,
   PRODUCT_MICRONUTRIENTS, MICRONUTRIENTS, MICRONUTRIENT_NORMS, ENERGY_NORMS.
3. Наполнить каталог блюд (RECIPES, RECIPE_INGREDIENTS, совместимость с диетами/аллергенами).
4. Ядро (Go): фильтрация → жадный подбор → балансировка по КБЖУ → проверка коридора →
   расчёт недобора микронутриентов. По АЛГОРИТМ_блок-схема.svg и МАТМОДЕЛЬ.txt.
5. API ядра по КОНТРАКТ_API.md.
6. Интеграция с price-service (опциональная вилка цены).
7. Доработка price-service (min/max + retailer_slug).
8. Фронтенд (React).
