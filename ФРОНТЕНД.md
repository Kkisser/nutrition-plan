# Спецификация фронтенда (PWA)

**Дата:** 19 мая 2026
**Назначение:** зафиксировать решение использовать PWA как способ доставки
React/TypeScript-фронта. Дополнение к `README.md` (где зафиксирован стек
«React / TypeScript») и `КОНТРАКТ_API.md` (где описан стык с ядром).

---

## 1. Решение

Фронт реализуется как **Progressive Web App** на стеке
**React 18 + TypeScript + Vite + vite-plugin-pwa**.

Обоснование:
- Установка на iOS/Android без публикации в App Store / Google Play —
  снимает блокер срока защиты ВКР.
- Офлайн-просмотр уже сформированного плана питания (план = недельный
  снимок, не требует постоянной сети).
- Единый кодовый стек React/TS уже зафиксирован в README; PWA — это
  способ доставки, а не альтернативный стек.

---

## 2. Технологический стек

| Слой | Выбор | Версия | Назначение |
|------|-------|--------|------------|
| Фреймворк | React | 18.x | UI |
| Язык | TypeScript | 5.x | типизация |
| Сборка | Vite | 5.x | bundler + dev server |
| PWA | vite-plugin-pwa | 0.20.x | manifest + Workbox-генератор Service Worker |
| Роутинг | React Router | 6.x | страницы |
| Состояние | React Query (TanStack Query) | 5.x | кеш API + повторы |
| Локальное хранение | IndexedDB через idb-keyval | 6.x | последний план, профиль |
| HTTP | fetch + React Query | — | без axios |
| Стили | CSS Modules или Tailwind | — | решение по код-стайлу команды |
| Тесты | Vitest + React Testing Library | — | unit-тесты компонентов |

Никаких тяжёлых фреймворков (Redux, MobX) — React Query закрывает
кеш-сервер-стейт, useReducer хватает для UI-стейта.

## 3. PWA-обёртка

### 3.1. manifest.json
- `name`: «Персональный план питания»
- `short_name`: «План питания»
- `start_url`: `/`
- `display`: `standalone` (без браузерной оболочки)
- `theme_color`: согласуется с дизайном
- `icons`: 192×192, 512×512, maskable

### 3.2. Service Worker
Генерируется через `vite-plugin-pwa` (`workbox` под капотом):
- **Precache** — App Shell: HTML, JS-bundle, CSS, иконки, шрифты.
- **Runtime cache**:
  - `GET /api/last-plan` → `StaleWhileRevalidate` (показать кэшированный
    план мгновенно, обновить в фоне).
  - `POST /plan`, `POST /pricing` → НЕ кэшировать (только online).
  - Изображения блюд → `CacheFirst` с лимитом 50 файлов.

### 3.3. HTTPS
- Прод — обязательно HTTPS (требование PWA).
- Локальная разработка — `localhost` исключение.

## 4. Офлайн-стратегия

Минимально (на старте):
1. App Shell — кэшируется при первом заходе.
2. Последний полученный план — пишется в IndexedDB при успешном
   `POST /plan`.
3. При офлайне:
   - роут `/plan` читает из IndexedDB, показывает пометку «офлайн».
   - роут `/replan` (формирование нового) показывает заглушку «нет сети,
     откройте план», кнопка переформирования недоступна.

Развитие (после защиты):
- Список покупок офлайн с отметкой «приобретено» — синхронизация при
  возвращении сети.
- Push-уведомления о готовности нового плана.

## 5. Структура папок

```
web/
├── public/
│   ├── manifest.webmanifest
│   └── icons/                   # 192, 512, maskable
├── src/
│   ├── main.tsx                 # точка входа
│   ├── App.tsx                  # роуты
│   ├── api/                     # клиент к core (POST /plan, /pricing, /health)
│   │   ├── client.ts            # fetch + react-query
│   │   ├── types.ts             # типы по КОНТРАКТ_API.md §2
│   │   └── persist.ts           # IndexedDB слой
│   ├── pages/
│   │   ├── Login.tsx            # email + password
│   │   ├── Survey.tsx           # анкета профиля + активность (Q1/Q3/Q4)
│   │   ├── Plan.tsx             # главный экран — план недели
│   │   ├── DishCard.tsx         # карточка блюда
│   │   ├── Replace.tsx          # экран замены блюда
│   │   ├── Shopping.tsx         # список покупок
│   │   └── Pricing.tsx          # оценка стоимости (выбор магазина)
│   ├── components/              # переиспользуемые UI-блоки
│   ├── hooks/                   # useProfile, usePlan, useShopping
│   └── lib/                     # вспомогательные функции
├── vite.config.ts               # + vite-plugin-pwa
├── package.json
└── tsconfig.json
```

## 7. Что НЕ делаем (зафиксировано)

- Нативные приложения (iOS/Android Swift/Kotlin) — за рамками ВКР.
- Серверный рендеринг (Next.js, Remix) — избыточно для размера задачи.
- Платный CDN, Sentry, аналитика — за рамками ВКР.
- Push-нотификации — направление развития (§16 ФУНКЦИОНАЛ.md).

---

## 8. Когда начинать

После закрытия шага 6 (HTTP-клиент к price-service) и опционально
аутентификации. По очерёдности README — это **шаг 8** очерёдности
разработки.

До старта фронта команда должна:
1. Утвердить дизайн-макеты (или использовать минималистичный без
   дизайнера — на защите визуальная отделка не критична).
2. Решить про CSS — Tailwind или CSS Modules.
3. Решить про деплой — Vercel/Netlify/GitHub Pages для PWA-демо
   на защиту.
