# ПРИНЯТЫЕ РЕШЕНИЯ

## РЕШЕНИЕ №1. Учёт микронутриентов в плане питания

**Дата:** 8 мая 2026
**Источник постановки вопроса:** замечание научрука [Т 0:53:01], исследовательская задача 1 в файле замечаний.
**Статус:** 🟢

### Суть решения
Микронутриенты в работе **учитываем** через механизм **переноса недобора на следующую неделю** (Вариант В1 — без отметок пользователя о реально съеденном).

### Логика
1. План формируется на неделю по существующей схеме (КБЖУ + диета + аллергии).
2. По итогу недели для каждого микронутриента *i* из перечня МР 2.3.1.0253-21 рассчитывается среднесуточное фактическое потребление *N_факт_i* и сравнивается с нормой *N_норма_i* для возрастно-половой группы пользователя.
3. Если *N_факт_i < N_норма_i*, недобор *Δ_i = N_норма_i − N_факт_i* сохраняется в БД.
4. При формировании плана на следующую неделю целевая величина для нутриента *i* корректируется по формуле:

   *N_цель = min(N_норма + Δ/7, N_ВДУ)*

   где
   - *N_норма* — суточная норма по МР 2.3.1.0253-21;
   - *Δ = max(0, N_норма − N_факт)* — недобор за прошлую неделю;
   - *N_ВДУ* — верхний допустимый уровень потребления (источники — см. Решение №2).

5. Долг не накапливается: на следующей итерации учитывается только недобор последней недели.

### Что НЕ делаем
- В основную штрафную функцию F микронутриенты **не включаются** (формула F остаётся без изменений).
- Веса для микронутриентов **не вводятся**.
- Выходной отчёт пользователю «дефицит того и того» **не делается**.
- Ежедневные отметки «съел / не съел» **не делаются**.

### ⚠️ Обоснование отхода от буквальной формулировки научрука (готовый ответ на защите)

Научрук на встрече [Т 0:53:01] просила добавить микронутриенты **в основную формулу F**. Решение №1 реализует **цель** этого замечания (обеспечение баланса по микронутриентам), но **другим механизмом** — не включением в F. Это осознанный, согласованный командой отход. Аргументация для защиты:

- В перечне МР 2.3.1.0253-21 порядка **29 микронутриентов**. Прямое включение в F превращает функцию из ~4 критериев (калорийность, белки, жиры, углеводы) в ~33 критерия.
- Каждому критерию в штрафной функции требуется **весовой коэффициент**. Обосновать 29 весов нормативным источником невозможно — таких источников не существует; произвольные веса делают модель недоказуемой.
- Множество блюд, одновременно закрывающих 29 нормативов за неделю, практически пусто → оптимизатор систематически не находит допустимого решения → модель теряет работоспособность.
- Поэтому баланс по микронутриентам обеспечивается **проверкой по факту сформированного плана и компенсацией недобора на следующей неделе**: F остаётся устойчивой и доказуемой, а цель замечания (сбалансированность по микронутриентам) достигается. Это инженерно более надёжное решение той же задачи.

Формулировка для защиты: «Замечание о включении микронутриентов в формулу было учтено по существу. Прямое включение 29 нутриентов в F делает модель неустойчивой и недоказуемой из-за необоснованных весов, поэтому проверка сбалансированности и компенсация недобора реализованы отдельным механизмом — цель достигается, модель остаётся корректной.»

### Где править в дипломе
- **Раздел 1** — добавить определение сбалансированного питания через цитату МР 2.3.1.0253-21.
- **Раздел 5 (Постановка задачи)** — удалить строку 264: «витамины, минералы и иные микронутриенты в настоящей работе не учитываются».

- **Раздел 7 (Сущность, To Be)** — добавить пару предложений про фиксацию недобора и компенсацию на следующей неделе.
- **Раздел 8 (Мат.модель)** — новый подраздел в конце с двумя формулами (покрытие + скорректированная целевая).
- **Раздел 10 (Алгоритм)** — два дополнительных блока в основной блок-схеме («Загрузка остатка дефицита», «Сохранение остатка дефицита»).

### Способ расчёта нутриентов в блюде

Блюдо состоит из продуктов (рецепт). Содержание любого нутриента в блюде вычисляется как взвешенная сумма содержаний по продуктам:

*N_блюдо_i = Σ (m_продукт_j × N_продукт_j_i / 100)*

где *m_продукт_j* — масса продукта *j* в граммах по рецепту, *N_продукт_j_i* — содержание нутриента *i* в продукте *j* в расчёте на 100 г съедобной части (по справочнику Скурихина-Тутельяна).

### Ограничения метода

В дипломе фиксируем как осознанные ограничения:
- **Расчёт ведётся по сырому весу продуктов** (без учёта потерь при кулинарной обработке). Для большинства макронутриентов потери незначительны; для термолабильных витаминов (C, B1) возможна систематическая переоценка фактического содержания. Учёт коэффициентов кулинарной обработки — направление развития работы.
- **Содержание нутриентов в продуктах** усреднённое по выборке справочника; индивидуальная вариативность по партии конкретного производителя не учитывается.

## РЕШЕНИЕ №3. Структура опросника пользователя (входной формы)

**Дата:** 8 мая 2026

**Статус:** 🟢

### Суть решения

Опросник пользователя разбивается на **6 логических блоков**. Заполняется при регистрации, потом редактируется в личном кабинете. Логика — от обязательного к опциональному.

### Состав блоков

**Блок 1. Физиологические данные** (обязательно — без них нельзя посчитать норму)
- пол (м/ж)
- возраст (лет)
- рост (см)
- масса тела (кг)
- уровень активности — ⚠️ **Изменено решением по функционалу №14.** Определяется не прямым выбором из 5 категорий, а **мини-анкетой** (дерево решений), относящей пользователя к одной из **4 групп КФА** по МР 2.3.1.0253-21. Энергетическая норма берётся из таблиц МР 2.3.1.0253-21; формула Миффлина — Сан Жеора не используется.

**Блок 2. Цель** (обязательно — определяет, в какую сторону смещаем калораж от нормы)
- снижение веса
- поддержание веса
- набор массы

**Блок 3. Тип питания** (обязательно, по умолчанию «Без ограничений»)
- классическое (без ограничений)
- кетодиета
- вегетарианство
- веганство
- палео
- пост (религиозный)

**Блок 4. Лечебный стол** — ⚠️ **Изменено решением по функционалу №10.** Поле «Лечебный стол» из опросника **полностью убрано** (а не «по умолчанию Нет»). Полноценная поддержка лечебных диет не предусмотрена; система столов Певзнера (МР 2.3.1.1915-04) — только направление развития, без поля в опроснике.

**Блок 5. Ограничения** (опционально)
- пищевые аллергии (множественный выбор: молоко, яйца, рыба, глютен, арахис, кунжут, моллюски, соя, орехи)
- индивидуально исключаемые продукты (свободный ввод)

**Блок 6. Структура приёмов пищи** (обязательно)
- активные приёмы пищи (множественный выбор: завтрак, обед, ужин, перекус)
- количество персон для приготовления (число)

- ❌ магазины (по решению научрука «магазины и наличие убраны»)

> Эти 15 решений — **самые свежие**. При расхождении с более ранними записями журнала приоритет у них (см. «ПРИОРИТЕТ АКТУАЛЬНОСТИ» в шапке).

### Решение по функционалу №1. Семейный режим
Блок 6 опросника — «количество персон» (число); ингредиенты в списке покупок умножаются на N, КБЖУ считается на одного. Плюс кнопка «объединить меню» (общий список покупок для двух пользователей с готовыми планами). Каждый пользователь — отдельная учётка. Убрано: 4 режима добавления участников, разрешение конфликтов аллергий, аватары, фильтрация по участникам.

### Решение по функционалу №2. Своё блюдо — убрать полностью
Каталог только курируемый администратором. Раздел 6.4 v3 и позиция в оценке сложности (раздел 12 v3) удаляются. Пользовательские рецепты — направление развития.

### Решение по функционалу №3. Кастомизация рецепта
На карточке блюда остаются: кнопка «Заменить» (процедура замены, раздел 10.4 — поиск по предварительно отфильтрованной базе) и контрол «Количество порций» (±, линейный пересчёт). Раздел 4.3 v3 (удаление/замена/изменение ингредиента) и позиция в оценке сложности удаляются.

### Решение по функционалу №4. Закреплённые блюда + fallback
Можно закрепить любое количество блюд в любых приёмах пищи; они не подбираются оптимизатором, целевые показатели свободных приёмов пересчитываются (из дневного целевого числа вычитаются закреплённые). Fallback: если коридор КБЖУ/нормы микронутриентов не выдержан — предупреждение пользователю (снять закрепления / принять с отклонением / переформировать). Слово «слот» → «приём пищи». **Затрагивает матмодель (раздел 8) и алгоритм (раздел 10).**

### Решение по функционалу №5. «По мере необходимости» — убрать полностью
Раздел 11.4 v3 и пункт в пакетных действиях (6.3 v3) удаляются. Покрывается закреплением (№4) и заменой (№3).

### Решение по функционалу №6. Цена и вилка [мин; макс]
Опциональная функция. Источник цен — агрегатор Edadil. Пользователь выбирает один магазин из доступных; не выбрал — плашка «Выберите магазин»; нажал «Пропустить» — вилка не показывается (на выходе только план + список покупок). Выбрал — вилка [мин; макс] по этому магазину. Геопривязка и проверка наличия не используются. Доработка `price-service`: добавить `max_total_price` в `/estimate` (сейчас только `total_price` как минимум). **Уточняет строку про цену в разделе 5.**

### Решение по функционалу №7. Целевая калорийность и её редактирование
Целевая калорийность = норма из таблиц МР 2.3.1.0253-21 (по полу, возрасту, группе КФА — см. Решение по функционалу №14) с поправкой на цель: дефицит −15% / удержание 0 / профицит +15% → дневное целевое число, к которому план стремится каждый день. Ручная правка пользователем — в пределах ±15% от нормы МР (система останавливает на границе). Отдельной «недельной цели» как параметра нет. Редактирование целей отдельного приёма пищи/дня недоступно (раздел 3.7 v3 в этой части убирается). Микронутриенты (Решение №1 журнала) считаются на недельном окне — отдельный механизм. **Затрагивает матмодель (раздел 8).**

### Решение по функционалу №8. Нутриционный фокус — фильтр
Фильтр «Нутриционный фокус» (мультивыбор: высокий протеин, низкоуглеводные, низкие жиры, высокая клетчатка, низкий натрий) в каталоге и в окне «Заменить».
