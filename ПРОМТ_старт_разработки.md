ТЕХНОЛОГИЧЕСКИЙ СТЕК (зафиксирован, не менять без явного запроса):
- Ядро формирования плана — Go
- Загрузка справочных данных (Скурихин-Тутельян) — Python
- Хранилище — PostgreSQL
- Кэш — Redis
- Фронтенд — React / TypeScript
- price-service — отдельный готовый Go-сервис (уже работает, дорабатывать не надо)

ЖЁСТКИЕ ПРИНЦИПЫ (нарушение = переделка, на защите завалят):
- Энергонорма берётся ТОЛЬКО из таблиц МР 2.3.1.0253-21 по (пол, возраст,
  группа КФА I–IV). Формула Миффлина — Сан Жеора НЕ используется нигде.
- Группа КФА определяется мини-анкетой (дерево решений), не прямым выбором.
- Бюджета НЕТ. Выбора магазина как параметра планирования НЕТ. Проверки
  наличия товаров НЕТ.
- Цена — опциональная вилка [min; max], считается ПОСЛЕ плана через
  price-service, по запросу. Цены в нашу БД НЕ сохраняются как справочник.
- Микронутриенты НЕ входят в штрафную функцию F. Отдельный механизм:
  по итогам недели считается недобор, переносится на следующую неделю
  (см. МАТМОДЕЛЬ.txt раздел 7).
- Пользовательские рецепты, лайки/дизлайки, пропуск приёма пищи — НЕ делаем.
- Каталог блюд — только курируемый, наполняется администратором.
- Слово «слот» не используется — «приём пищи».

АЛГОРИТМ ЯДРА (кратко, детали в МАТМОДЕЛЬ.txt и блок-схеме):
двухфазный. Фаза 1 — жадный подбор блюд по приёмам пищи (минимум штрафа F
по КБЖУ). Фаза 2 — локальный поиск заменами для балансировки суммарных
дневных КБЖУ к целевому числу. Затем расчёт недобора микронутриентов,
проверка коридора, при невыполнении — предупреждение пользователю
(снять закрепления / принять с отклонением / переформировать).

ВЗАИМОДЕЙСТВИЕ С price-service (уже готов и доработан):
- POST http://localhost:8085/estimate, вход — агрегированный items[]
  (формат совпадает с shopping_list ядра: ingredient_name, amount, unit∈{g,ml,pcs}).
- Возвращает min_total_price / max_total_price / price_range.
- Выбор магазина — опциональное поле selected_shop_uuid (Edadil shopUuid
  конкретной точки), НЕ retailer_slug. Не передан → вилка по ближайшим
  магазинам. Передан → по одному магазину.
- Если пользователь пропустил оценку цены — price-service не вызывать.
- Детали в docs/КОНТРАКТ_API.md раздел 3.

РЕКОМЕНДОВАННАЯ ОЧЕРЁДНОСТЬ (из README, можно корректировать):
1. PostgreSQL DDL по docs/СХЕМА_БД.md
2. Python-загрузчик справочника продуктов (Скурихин-Тутельян) → PRODUCTS,
   PRODUCT_MICRONUTRIENTS, MICRONUTRIENTS, MICRONUTRIENT_NORMS, ENERGY_NORMS
3. Наполнение каталога блюд (RECIPES + состав + совместимость диет/аллергенов)
4. Ядро (Go): фильтрация → жадный подбор → балансировка → проверка коридора
   → недобор микронутриентов
5. API ядра по docs/КОНТРАКТ_API.md
6. Интеграция с price-service
7. Фронтенд (React)
